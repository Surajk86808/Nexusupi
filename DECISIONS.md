Question 1:
The credit system uses a transaction ledger because correctness under concurrency matters more than convenience of a single mutable number. A balance column looks simple, but it creates a classic read modify write race: request A and request B both read balance 20 at the same time, each decides there are enough credits for a 15 credit operation, and both write back results that appear valid in isolation while producing an invalid final state. A ledger avoids that overwrite pattern entirely because every event is an append, not an in place mutation. Each grant or deduction becomes an immutable row with amount, reason, actor, and timestamp, and current balance is derived from full history using SUM. That introduces a modest aggregation cost, but it removes hidden state transitions and gives deterministic reconstruction of how balance reached its current value. The ledger also provides auditability, easier incident investigation, and reversible business logic through compensating entries rather than destructive edits, which is a safer operational model.

Question 2:
The simultaneous deduction problem is handled with pessimistic locking using SELECT FOR UPDATE inside a single transaction boundary. When two requests target the same organisation at nearly the same moment, one request acquires row locks first and computes balance from the locked ledger rows. The second request blocks at the lock point and cannot read a stale pre deduction view for decision making. If the first request deducts and commits, the second request resumes, recomputes from the now updated ledger, and may fail with InsufficientCreditsError before inserting anything. This serialises decision critical paths at the database layer and prevents double spend behavior without requiring complex retries in clients or API handlers. Optimistic concurrency with version columns was considered but rejected for this path because it shifts conflict handling into retries that every caller must implement consistently, which tends to produce partial implementations over time. Here, centralising contention control in SQL keeps the behavior deterministic and locally testable under concurrent load.

Question 3:
Credits are deducted before enqueueing and processing because the operation is priced as access to compute capacity, not strictly successful output delivery. If the worker fails after deduction, the system preserves the debit and represents execution state explicitly in job status as pending or failed. It does not auto refund failed jobs. The reason is practical and consistency driven: automatic refunds add another transaction branch that can fail independently, and then the platform has to reconcile a larger set of broken states such as deducted but not refunded, refunded twice, or refunded after partial downstream success. By avoiding implicit compensation, the system remains predictable: one request creates one debit and one job record trajectory. The user can inspect status through GET /api/jobs/{job_id}, and operational support can decide whether to issue manual credit adjustments when needed. ARQ timeout is set to 300 seconds so long hanging tasks converge to failure semantics instead of indefinite pending, which improves observability and prevents jobs from disappearing into ambiguous limbo.

Question 4:
Idempotency is implemented in the credit transaction service layer and enforced in the database schema through a UNIQUE constraint on idempotency_key scoped to the relevant transaction identity. The service first checks for an existing transaction with the same key for the same organisation and returns it immediately when present, which handles normal retries efficiently. The stronger safety property comes from the constraint, not the precheck, because concurrent duplicates can bypass application checks if they race. In that race, both requests can reach insert, but only one row can persist with the unique key. The loser receives a constraint violation, which the service catches and treats as duplicate completion by loading and returning the already committed transaction instead of charging again. This means idempotency is atomic at storage level and does not depend on process local locks, sticky sessions, or timing luck. The implementation lives where business invariants are applied, near deduction logic, while the database remains final authority for conflict resolution under true concurrency.

Question 5:
At ten times current throughput, the first major pressure point is balance computation cost, because every charge path currently performs SUM over the organisation ledger before writing a deduction. As ledgers grow to thousands of rows per tenant, that aggregation becomes increasingly expensive and can dominate request latency, especially during bursty periods with many concurrent deductions against active organisations. The remedy is to keep the ledger as source of truth while introducing a transactional balance cache updated in the same transaction as each ledger insert. That can be a dedicated table keyed by organisation_id, or a Redis backed mirror with careful write through semantics and fallback to SQL reconciliation. Reads then use the cached value instead of scanning full history for each request. The second likely issue is database connection pool saturation, which can cascade into queueing delays and timeout spikes. Addressing that requires calibrated pool sizing, bounded request concurrency, and explicit queue timeouts so overload fails fast and predictably instead of accumulating indefinite waits that degrade the entire API.
